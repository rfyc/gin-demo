# 开发工作流

> 本文件是 [common/git-workflow.md](./git-workflow.md) 的扩展，涵盖 git 操作之前的完整功能开发流程。

功能实现工作流描述了完整的开发流水线：调研、规划、TDD、代码审查，最后提交到 git。

## 功能实现工作流

0. **调研与复用** _(任何新实现前的必要步骤)_
   - **优先搜索 GitHub 代码：** 在编写任何新代码前，运行 `gh search repos` 和 `gh search code` 查找已有实现、模板和模式。
   - **其次查阅库文档：** 使用 Context7 或主要厂商文档确认 API 行为、包用法和版本特定细节。
   - **Exa 作为兜底：** 仅在前两者不足时，使用 Exa 进行更广泛的网络调研或发现。
   - **检查包注册表：** 在编写工具代码前，搜索 npm、PyPI、crates.io 等注册表。优先使用经过验证的库，而非自行实现。
   - **寻找可适配的实现：** 查找能解决 80% 以上问题的开源项目，可以 fork、移植或封装使用。
   - 满足需求时，优先采用或移植已验证的方案，而非从头编写新代码。

1. **先规划**
   - 使用 **planner** agent 创建实现计划
   - 编码前生成规划文档：PRD、架构设计、系统设计、技术文档、任务清单
   - 识别依赖和风险
   - 分阶段拆解

2. **TDD 方法**
   - 使用 **tdd-guide** agent
   - 先写测试（RED）
   - 实现代码使测试通过（GREEN）
   - 重构（IMPROVE）
   - 验证覆盖率 80% 以上

3. **代码审查**
   - 编写代码后立即使用 **code-reviewer** agent
   - 解决 CRITICAL 和 HIGH 问题
   - 尽可能修复 MEDIUM 问题

4. **提交与推送**
   - 详细的提交信息
   - 遵循 conventional commits 格式
   - 提交信息格式和 PR 流程参见 [git-workflow.md](./git-workflow.md)

5. **审查前检查**
   - 验证所有自动化检查（CI/CD）通过
   - 解决所有合并冲突
   - 确保分支已同步目标分支
   - 通过以上检查后才请求审查
