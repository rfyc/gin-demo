# 代码审查规范

## 目的

代码审查在合并前确保质量、安全性和可维护性。本规则定义何时以及如何进行代码审查。

## 何时审查

**必须触发审查的情况：**

- 编写或修改代码后
- 向共享分支提交前
- 修改安全敏感代码时（认证、支付、用户数据）
- 进行架构变更时
- 合并 Pull Request 前

**审查前置要求：**

请求审查前确保：

- 所有自动化检查（CI/CD）通过
- 合并冲突已解决
- 分支已同步目标分支的最新内容

## 审查清单

标记代码完成前：

- [ ] 代码可读，命名清晰
- [ ] 函数职责单一（< 50 行）
- [ ] 文件内聚（< 800 行）
- [ ] 无深度嵌套（> 4 层）
- [ ] 错误已显式处理
- [ ] 无硬编码密钥或凭据
- [ ] 无 console.log 或调试语句
- [ ] 新功能有对应测试
- [ ] 测试覆盖率达到 80% 最低要求

## 安全审查触发条件

**遇到以下情况立即停止，使用 security-reviewer agent：**

- 认证或授权代码
- 用户输入处理
- 数据库查询
- 文件系统操作
- 外部 API 调用
- 密码学操作
- 支付或金融代码

## 审查严重程度

| 级别 | 含义 | 行动 |
|------|------|------|
| CRITICAL | 安全漏洞或数据丢失风险 | **阻塞** —— 合并前必须修复 |
| HIGH | Bug 或重大质量问题 | **警告** —— 合并前应修复 |
| MEDIUM | 可维护性问题 | **信息** —— 考虑修复 |
| LOW | 风格或次要建议 | **备注** —— 可选 |

## Agent 使用

代码审查使用以下 agent：

| Agent | 用途 |
|-------|------|
| **code-reviewer** | 通用代码质量、模式、最佳实践 |
| **security-reviewer** | 安全漏洞、OWASP Top 10 |
| **typescript-reviewer** | TypeScript/JavaScript 专项问题 |
| **python-reviewer** | Python 专项问题 |
| **go-reviewer** | Go 专项问题 |
| **rust-reviewer** | Rust 专项问题 |

## 审查工作流

```
1. 执行 git diff 了解变更内容
2. 先检查安全清单
3. 审查代码质量清单
4. 运行相关测试
5. 验证覆盖率 >= 80%
6. 使用对应 agent 进行详细审查
```

## 常见问题排查

### 安全

- 硬编码凭据（API 密钥、密码、token）
- SQL 注入（查询中字符串拼接）
- XSS 漏洞（未转义用户输入）
- 路径穿越（未净化的文件路径）
- 缺少 CSRF 保护
- 认证绕过

### 代码质量

- 函数过大（> 50 行）—— 拆分为更小的函数
- 文件过大（> 800 行）—— 提取模块
- 深度嵌套（> 4 层）—— 使用提前返回
- 缺少错误处理 —— 显式处理
- 可变模式 —— 优先使用不可变操作
- 缺少测试 —— 补充测试覆盖

### 性能

- N+1 查询 —— 使用 JOIN 或批量处理
- 缺少分页 —— 为查询添加 LIMIT
- 无界查询 —— 添加约束条件
- 缺少缓存 —— 对耗时操作添加缓存

## 审批标准

- **批准**：无 CRITICAL 或 HIGH 问题
- **警告**：仅有 HIGH 问题（谨慎合并）
- **阻塞**：发现 CRITICAL 问题
